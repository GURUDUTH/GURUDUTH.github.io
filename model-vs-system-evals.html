Model Evals vs System Evals for AIPM/PM

By Guruduth Rao

LLM OBSERVABILITY : Evaluations

Before we go into how and when should we integrate observability, let's understand why do we actually need observability, evaluation and metrics in order to test an AI application, and to understand that we need to look at the most common pain points that companies face while building AI based applications or application while in production that use large language models, and some of those pain points are:

  1. Model hallucination - Regardless of using proper prompting and frameworks the application still hallucinates.
  2. Manual evaluations are hard - improper exposure to Automated evaluation tools or services.
  3. Bad RAG - if user asks questions outside of the context that generation is improper and irrelevant.
  4. No proper visibility into Ai agents performance - you cant say if your agents is actually improving with every interaction or not.

Now these pain points can be taken care of by improving LLM workflows while in production and those improvements are usually around Prompt engineering, Context retrieval, Fine tuning, these workflow improvements are really important to understand the evaluations.

And you may ask why knowledge of these workflows are important? cause you might have certain evaluation metrics but there is things you need to do to improve that response from an LLM application before evaluating that response itself.

LLM model Evals VS LLM system Evals

LLM model evals are focused on the overall performance of the foundational models. for an example if we are to take two open source foundational models and see which performs the best and then Testing those two models with similar dataset across the two models and seeing how their metrics perform for a variety of tasks.

LLM system evals basically happens after you have selected a model and integrated into your system, LLM system evaluation is all about evaluating the components in your system that you have control over such as prompt templates, because you have choice over choosing which model you want to use for your system, but you can't really change that large language model because its more like a black box and it may also be a proprietary model that you are using, but once you have integrated into your system the next step is doing the system evaluation and making sure that you can optimise it as best as possible, then see how well does your inputs determine the output from the system.

Let's understand this with a simple example:

Say you are using a single foundational model in your system but you are trying out two different prompt template by testing the same data across those two templates and see performance metrics like precision and relevance.
